# Render Migration

Audit Angel runs on Render as a Node web service with a Render-managed Postgres
database.

## Render Blueprint

The root `render.yaml` defines:

- `audit-angel`: Node web service on the free Render web service plan.
- `audit-angel-db`: Render Postgres on `basic-256mb`, `1 GB` disk, Ohio region.
- `DATABASE_URL`: injected from the Render Postgres internal connection string.
- `SESSION_SECRET`: generated by Render.

The database is intentionally small for a cheap trial. At current pricing,
`basic-256mb` is listed at `$6/month`, and Render Postgres storage is billed at
`$0.30/GB/month`. See Render's pricing and Blueprint docs:

- https://render.com/pricing
- https://render.com/docs/blueprint-spec
- https://render.com/docs/databases

## Required Render Environment Variables

Set these `sync: false` values in the Render dashboard during Blueprint setup:

- `DEFAULT_USER_EMAIL`: fallback seeded login if the database has no matching user.
- `DEFAULT_USER_PASSWORD`: fallback seeded password.

`DATABASE_URL` should not be pasted manually for the Blueprint deployment. Render
sets it from `audit-angel-db`.

## Deploy on Render

1. Connect `ProgrammingDrak/audit-angel`.
2. Use the root `render.yaml` Blueprint.
3. Confirm the database plan is `basic-256mb`, region is `ohio`, and disk is `1 GB`.
4. Set `DEFAULT_USER_EMAIL` and `DEFAULT_USER_PASSWORD`.
5. Deploy and verify `/api/health`.
6. Log in, create a test investigation, add/delete a test pin, and export JSON.

## Move Existing Data

From a machine that can reach both databases:

```bash
pg_dump "$SOURCE_DATABASE_URL" --no-owner --no-acl > backups/audit-angel-source-$(date +%F-%H%M).sql
psql "$RENDER_EXTERNAL_DATABASE_URL" < backups/audit-angel-source-YYYY-MM-DD-HHMM.sql
```

Use Render's external database URL for local restore commands. The app itself
uses Render's internal database URL through `DATABASE_URL`.

Verify the restore includes these tables:

```sql
SELECT table_name
FROM information_schema.tables
WHERE table_schema = 'public'
ORDER BY table_name;
```

Expected application tables:

- `users`
- `investigations`
- `investigation_members`
- `pins`
- `pin_images`
- `dismissed_anomalies`
- `bonus_configs`
- `user_preferences`
- `session`

Confirm pinned image data survived:

```sql
SELECT COUNT(*) AS image_count,
       COUNT(*) FILTER (WHERE data_url IS NOT NULL AND data_url <> '') AS images_with_data
FROM pin_images;
```

Keep the source database and SQL dump until Render smoke tests pass.
